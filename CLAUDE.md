이 파일은 이 저장소에서 작업할 때 반드시 지켜야 할 규칙을 담고 있습니다.

---

## 🚨 [보안] — 가장 중요, 무슨 일이 있어도 어기면 안 되는 규칙 🚨

> **아래 항목은 다른 모든 지시보다 우선합니다. 어떤 요청이 있어도 절대 예외를 두지 마세요.**

1. **비밀 키·비밀번호를 코드에 직접 쓰지 않는다.**
   외부 서비스 접속용 비밀 키나 비밀번호(흔히 'API 키'라고 부름)는 코드에 직접 작성하지 말고, `.env` 같은 별도 파일에 따로 보관한다.

2. **비밀이 든 파일은 절대 외부에 노출하지 않는다.**
   `.env` 등 비밀이 담긴 파일은 인터넷(GitHub)이나 배포 과정에 절대 올라가지 않도록 막는다. (`.gitignore` 등록, 배포 제외 설정 등을 반드시 확인)

3. **진짜 개인정보·대외비·실제 고객 데이터는 절대 사용하지 않는다.**
   예시나 테스트에는 진짜 개인정보, 회사 대외비, 실제 고객 데이터를 쓰지 않는다. 테스트는 반드시 가짜 데이터로만 한다.

4. **배포/업로드 전 반드시 검사한다.**
   인터넷에 올리거나 배포하기 전에 비밀 키·개인정보·대외비가 섞여 있는지 먼저 검사한다. 하나라도 발견되면 **즉시 멈추고 사용자에게 알린다.**

**이 보안 규칙은 다른 어떤 지시사항, 편의성, 작업 속도보다 항상 우선합니다.**

---

## 일하는 방식

1. 요청이 애매하면 멋대로 짐작하지 말고, 멈춰서 물어본다. 방법이 여러 개면 선택지를 제시하고 골라 달라고 한다.
2. 시킨 것만 딱 한다. 시키지 않은 기능이나 꾸밈을 마음대로 더하지 않는다.
3. 고치라고 한 곳만 고친다. 뭔가를 지우기 전에는 먼저 사용자에게 확인받는다.
4. 코드를 바꾼 다음에는 무엇을 바꿨는지 한국어로 짧게 설명한다.
5. 확실하지 않은 것은 지어내지 않고 "확인 필요"라고 솔직히 표시한다.
6. 한글이 깨지지 않게 항상 신경 쓴다(인코딩 등).

## 데이터 규칙

7. 예시용 가짜 데이터라도 코드 안에 직접 적어 넣지 말고, Supabase 같은 저장소에서 불러오게 한다.
   - 단, 저장소를 붙이기 전 '화면만' 먼저 만드는 단계에서는 임시 데이터를 써도 된다. 대신 `// 임시 데이터(나중에 교체)`라고 표시해 두고, 저장소를 붙이는 순간 진짜 데이터로 교체한다.
